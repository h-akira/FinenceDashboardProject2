# Enhancement ワークフロー詳細

## フェーズ1: 起票・計画

### 1.1 要件の確認

開発者の指示を受けたら、まず不明点がないか確認する。

- 指示の意図・背景が不明確な場合は、enhancement ファイル作成前に開発者に質問する
- 要件の範囲・優先度・制約条件など、曖昧な点があれば先に解消する
- 不明点がなければそのまま次に進む

### 1.2 Enhancement ファイル作成

1. `enhancements/` 内の既存ファイルを確認し、最大番号 + 1 で採番する
2. `enhancements/XXX_template.md` を読み、テンプレートに従って新規ファイルを作成する
   - ファイル名: `XXX_（概要を表すスネークケース）.md`
   - テンプレート内の AI instruction コメントに従う
   - ステータスは「未着手」で開始
3. 開発者の指示から要件を整理し、以下を埋める:
   - 種別（仕様追加 / 改善 / 改良）
   - 概要・背景・動機
   - 要件（具体的な箇条書き）
   - 影響範囲
   - 実現案（技術的な設計）

### 1.3 計画時のドキュメント・実装確認

計画の精度を上げるため、以下を必ず確認する:

- `docs/user_stories.md` — 関連するユーザーストーリー
- `docs/units_definition.md` — 影響ユニットの責務確認
- `docs/units_contracts.md` — ユニット間契約（命名規約、エクスポート等）
- `docs/openapi_main.yaml` — API 定義の現状
- `docs/technical_policies.md` — 技術方針への適合確認
- 影響を受ける実装ファイル — 現状の実装を把握した上で設計する

### 1.4 計画の提示とレビュー

- enhancement ファイルの内容を開発者に提示し、レビューを受ける
- フィードバックを反映して計画を確定させる

## フェーズ2: ドキュメント更新

実装に入る前に、共有ドキュメント（`docs/`）への更新が必要かを確認する。

### 2.1 ドキュメント更新チェック

以下の観点で各ドキュメントを確認する:

| ドキュメント | 更新が必要なケース |
|-------------|------------------|
| `docs/openapi_main.yaml` | API エンドポイント・スキーマの追加/変更 |
| `docs/user_stories.md` | ユーザーストーリーの追加/変更 |
| `docs/units_definition.md` | ユニットの責務変更 |
| `docs/units_contracts.md` | ユニット間契約の変更 |
| `docs/technical_policies.md` | 技術方針の変更（まれ） |

### 2.2 ドキュメント更新の実施

- `docs/` の変更には開発者の許可が必要
- 更新前に開発者に変更内容を説明し、承認を得る
- ドキュメント内の AI instruction コメントに従う

### 2.3 仕様変更通知の作成

`docs/` に変更を加えた場合、必ず `docs/history/` に変更通知ファイルを作成する。

1. `docs/history/` 内の既存ファイルを確認し、最大番号 + 1 で採番する
2. `docs/history/XXX_template.md` を読み、テンプレートに従って作成する
   - ファイル名: `XXX_（概要を表すスネークケース）.md`
   - 対象ファイル: 変更したドキュメントの一覧
   - 変更内容: 何をどう変更したか
   - 変更理由: enhancement の背景と紐づけて説明
   - 影響を受けるユニット: 各ユニット（Infra, Frontend, Backend/main, CI/CD）への影響と必要な対応

### 2.4 コミット

- ドキュメント更新と変更通知の作成が完了したら、開発者にコミットしてよいか確認する
- 開発者が承認した場合は git commit を行う
- 開発者が不要と判断した場合はコミットせずにフェーズ3に進む

## フェーズ3: 実装

### 3.1 実装の進め方

- 独立したユニットごとに順番に実装する
- 各ユニットのサブモジュール内 `docs/` がある場合はそれにも従い、必要に応じて更新する
- 他のサブモジュールの内部実装には原則依存しない
- `docs/` の定義（API 契約・命名規約・通信仕様等）に厳密に従う

### 3.2 実装順序の目安

一般的な順序（依存関係による）:

1. **バックエンド**: API・ロジックの変更
2. **フロントエンド**: UI・ページの変更
3. **インフラ**: リソースの追加・変更（必要な場合）
4. **CI/CD**: パイプラインの変更（必要な場合）

### 3.3 テストの実施

- 各ユニットの実装後にユニットテストを実行する
- 既存テストの更新が必要な場合は更新する
- 新規テストが必要な場合は追加する

## フェーズ4: 完了

### 4.1 Enhancement ファイルの更新

- 「対応」セクションに実装内容を追記する
  - 採用した実現案
  - 変更したファイル・メソッドの列挙
- ステータスを「完了（検証待ち）」に更新する

### 4.2 検証後

- 開発者による検証が完了したら、ステータスを「完了（検証済み）」に更新する
